QSense · On-device predictive maintenance

A fault alert becomes a grounded fix — on the device, in seconds.

An MQTT fault alert arrives, relevant failure modes are retrieved from an on-device knowledge base, a Qualcomm on-device LLM turns them into ranked causes & fixes, and the operator resolves it — no cloud round-trip. v2 adds camera-based part inspection over the same MQTT fabric.

KMP + Compose Multiplatform GenieX on-device LLM (GGUF) MQTT 5 · QoS 1 RAG grounding + fallback Vision v2 phone ↔ PC

02 · Architecture

Clean-architecture KMP: logic is pure, SDKs are adapters

shared / commonMain pure Kotlin

  • Domain models + service interfaces: TextGenerator, MqttGateway, Clock
  • Use cases: GenerateDiagnosis, IngestAlerts, PublishResolution
  • RAG: KnowledgeBase + InMemoryKnowledgeBase
  • Prompt builder + tolerant JsonDiagnosisParser
  • Compose UI + ViewModels + theme

shared / androidMain SDK adapters

  • GenieXTextGenerator → Qualcomm GenieX (GGUF)
  • HiveMqttGateway → HiveMQ MQTT 5 client
  • SystemClock, CameraX capture (vision)
  • Compose expect/actual for camera + image

androidApp thin shell

  • Builds the AppContainer once (manual DI — no Hilt/Koin)
  • Hosts the activity, loads Clash Display + Satoshi fonts
  • Adaptive launcher icon (the QSense badge)

server / Python · vision v2

  • paho-mqtt subscriber on qsense/vision/request
  • Pillow annotate → RTMDet (ONNX) detection
  • Publishes annotated image + diagnosis back
  • Pure handler() → unit-testable without a broker

Why: the domain never imports an Android SDK, so use cases and ViewModels are unit-testable on the JVM with fakes. Each vendor SDK sits behind one interface and can be swapped without touching logic.

04 · The core

How the model works with RAG

The small on-device model isn't asked to invent from nothing — it's asked to select and adapt from failure modes already retrieved for that part. Retrieval is deterministic; the model does the ranking and rewriting; and if it stumbles, the retrieved knowledge is the answer.

1 · Retrieve InMemoryKnowledgeBase

Deterministic keyword match on the alert's part name ("blade", "fan", "bearing"…) → up to 6 KnowledgeEntry (symptom → cause → fix). No embeddings, no model. Falls back to generic motor knowledge if the part is unknown.

↓

2 · Ground DiagnosisPrompt

A fixed SYSTEM prompt (expert persona + strict JSON contract + worked example) plus a per-alert message carrying the machine/part, sensor readings, and the retrieved reference known issues. Instruction: “select the most relevant, order most→least likely, rewrite for this machine.”

↓

3 · Generate GenieX (GGUF)

Streams tokens under an ASCII-only GBNF grammar — the crash guarantee (invalid UTF-8 would abort the JNI layer). Target shape: {"causes":[{"cause":"…","fix":"…"}]}. Structure is elicited by the prompt, not forced by the grammar.

↓

4 · Parse tolerant

Strip markdown fences, isolate the {…} span, lenient-decode, cap to 5 causes → Diagnosis.

5 · Fallback RAG carries it

If the model throws or nothing parses into ≥1 cause, the retrieved entries become the diagnosis. The operator always gets grounded causes/fixes — never an empty panel.

Why this shape wins for a tiny model: select-and-adapt over retrieved facts is far easier than open-ended generation, and the fallback means correctness never depends on the LLM succeeding. RAG is both the grounding and the safety net.
05 · On-device findings

What we learned running an LLM on the phone

✕
Structured GBNF grammar to force JSON → masks nearly every token, sampler empties, stream aborts (-200101).
✕
Invalid UTF-8 from split multi-byte tokens → JNI NewStringUTF aborts the whole process.
✓
ASCII-only grammar ([\t\n\r\x20-\x7E]) → the crash guarantee; safe bytes only, structure left to the prompt.
✕
Qwen3-0.6B can't hold “emit only this JSON” across a full generation — stray token, then stops.

Why a small model — and the honest tradeoff

0.6B was chosen so the model fits the phone's memory/latency budget with no cloud. But 0.6B is below the threshold for reliable structured JSON. Net effect today: the RAG fallback usually carries the diagnosis — the app stays grounded and useful, but the LLM isn't yet doing the work it was designed for.

fix →
Side-load a 1B-class instruct GGUF (biggest quality jump, least code).
opt →
Or MediaPipe + Gemma-3 1B: drops the ASCII workaround & Snapdragon lock-in.
keep →
Keep the RAG fallback regardless — even 1B on-device isn't cloud-grade.

Verified by 23 automated tests (JVM host + Python) plus a live broker round-trip. The takeaway: fix the model size, not the runtime — the runtime was never the bottleneck.
